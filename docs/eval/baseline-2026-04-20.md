# jarvis-memory retrieval baseline — 2026-04-20

**Harness:** `python -m jarvis_memory.eval`
**Corpus:** `tests/eval_data/synthetic_corpus_v1.jsonl` (150 episodes, 10 topics, 5 group_ids)
**Queries:** `tests/eval_data/synthetic_queries_v1.jsonl` (50 queries)
**Qrels:** `tests/eval_data/synthetic_qrels_v1.jsonl` (50, 1–5 relevant per query)
**Search path measured:** composite `scored_search` pipeline — ChromaDB semantic similarity (sentence-transformers/all-MiniLM-L6-v2) → Neo4j metadata enrich → `score_results` (0.5 semantic + 0.3 recency + 0.2 importance × type-weight).
**Namespace:** isolated `:TestEpisode` Neo4j label + `jarvis_eval_v1` Chroma collection (torn down on exit).
**Committed artifacts:** `/tmp/baseline_report.json` (local-only; regenerate anytime via the CLI below).

## Reproduce

```bash
cd ~/Atlas/jarvis-memory
source .venv/bin/activate
python -m jarvis_memory.eval \
  --corpus tests/eval_data/synthetic_corpus_v1.jsonl \
  --queries tests/eval_data/synthetic_queries_v1.jsonl \
  --qrels tests/eval_data/synthetic_qrels_v1.jsonl \
  --k 1,3,5,10 --json --ingest-corpus-first
```

## Interpretation

**Top-1 is strong; mid-range is the weak spot.** P@1 = 0.74 and MRR = 0.825 mean the first hit is relevant for roughly three out of four queries — that's a useful signal-rich retrieval for the `wake_up` and `continue_session` flows, which only show the best few episodes. Recall@5 = 0.757 means about a quarter of the relevant set is still missed at k=5 even though the corpus is small and topically clustered. Recall@10 = 0.855 closes most of that gap, but the jump from 5 → 10 (7.57 → 8.55) tells us relevant-but-not-top-5 items are often buried just past the cutoff.

**Where composite scoring helps vs. hurts.** The recency component and persistent-type floor lift old [DECISION] and [PREFERENCE] episodes; the 20% importance weight makes high-importance [PLAN]/[DECISION] items dominate. The nDCG@3 dip (0.651, below nDCG@5 = 0.717) suggests the top-3 window ingests some distractors: either false positives from narrow-vocab topics (e.g., tiny overlaps with unrelated rooms) or composite-score ties pulling older but persistent items ahead of newer relevant ones. Those are the exact failure modes RRF and intent-classification (Run 3) are meant to fix.

**Baseline to beat in Run 3.** Any future search change (RRF, query intent classifier, multi-query expansion, typed-edge boosting) must tie or beat:
- **Recall@5 ≥ 0.757** (primary headline)
- **MRR ≥ 0.825**
- **nDCG@10 ≥ 0.758**

Regressions on P@1 or P@3 are also red flags — they signal the "right answer up top" property deteriorating.

## Caveats

- **Synthetic corpus.** Episodes and queries come from Claude Opus (4-6) generation keyed to 10 hand-picked project topics, with a small random-template tail for short-fallers. They mimic jarvis's shape (group_id, episode_type, tagged prose) but do not reflect real Alex-authored episodes. Run 2+ should harvest a small real-sample qrel set to cross-check.
- **Small scale.** n=50 queries means the confidence intervals around each number are wide (±5–10 pp). Single-run deltas under ~3 pp are noise; require ≥ 3 pp to call a win.
- **One model of search.** Only the production `scored_search` (Chroma → enrich → composite) is measured. `wake_up`-style retrieval has its own ranking and is not captured here — that's deferred per spec Q1.
- **Seed-documented, not seed-identical.** The on-disk corpus is the reproducible artifact. Re-running `scripts/gen_eval_corpus.py` against the same RNG seed will produce a different byte-identical output only on the template-path; the LLM path is sampled at temperature 0.3 and is approximately reproducible but not bit-identical.
